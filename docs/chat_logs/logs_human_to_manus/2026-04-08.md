**问题1：点击章没有反应**

在第 93-99 行的 useEffect 中：
```typescript
useEffect(() => {

    setExpandedCantos(new Set([cantoId]));
    setExpandedChapters(new Set([chapterId]));

```

这个 useEffect 每次 `showToc`、`cantoId` 或 `chapterId` 变化时都会执行，**它会覆盖用户的点击操作**！

1. 用户点击一个章，`expandedChapters` 被更新
2. 但如果此时 `chapterId` 发生变化（比如导航到其他章），这个 useEffect 就会再次执行

- 依赖数组改为 `[showToc]`

- 添加 `expandedChapters` 到依赖数组

## 会话 3

**用户消息：**
为什么还是没变化？你改的是新的组件吗，现在SB和BG都用的是新组件吗

**Manus 操作：**

我找到问题了！

**BGReadPage 没有传递 `cachedCantos` prop！**

在 BGReadPage 第 369-392 行，调用 SBTableOfContents 时：
```typescript
<SBTableOfContents
  bookType="bg"
  cantos={[]}
  chapters={chapters}
  cantoData={{ sections: allSections }}
  chapterId={chapterId}
  sectionIndex={sectionIndex}
  cantoId={null}
  // ... 其他props
  // 缺少 cachedCantos prop！
/>
```

而 SBReadPage 第 503-508 行有传递：
```typescript
<SBTableOfContents
  bookType="sb"
  cantos={cantos}
  chapters={chapters}
  cantoData={cantoData}
  cachedCantos={cachedCantos}  // ✓ 这个有传
  // ...
/>
```

但这还不是根本问题。根本问题是：

**`useSBCantoDataCache` 返回的 `cachedCantos` 是一个 `Set<number>`（篇ID的集合），而不是实际的数据！**

在 useData.ts 第 151 行：
```typescript
const [cachedCantos, setCachedCantos] = useState<Set<number>>(new Set());
```

这只是记录哪些篇已经被加载了，但没有返回实际的数据。

而在 SBTableOfContents 中，我们期望 `cachedCantos` 是：
```typescript
Record<number, { sections: Record<string, SBSection[]> }>
```

这是一个数据映射，不是一个 ID 集合。

修复方案：
改为返回 `Record<number, SBCantoData>` 类型：

```typescript
// 正确的实现
const [cachedCantos, setCachedCantos] = useState<Record<number, SBCantoData>>({});
setCachedCantos(prev => ({ ...prev, [cantoId]: data }));  // 存储实际数据
```

这样 SBTableOfContents 就能通过以下方式访问缓存数据：
```typescript
cachedCantos[canto.id]?.sections[String(ch.id)]
```

代码改动：
- useData.ts：修复 useSBCantoDataCache 的返回类型
- BGReadPage.tsx：在调用 SBTableOfContents 时添加缺失的 props

这个错误 `net::ERR_CONNECTION_RESET` 加上"连WiFi就能访问"，进一步确认了判断：

## 确定原因：**移动数据网络的 DNS 污染**

两个关键证据：
1. **iPhone（5G）**：Safari 显示"丢失网络连接" → DNS 污染
2. **vivo（移动数据）**：`ERR_CONNECTION_RESET` → 连接被重置，典型的 DNS 污染或 SNI 封锁
3. **vivo 连 WiFi 后正常** → WiFi 路由器的 DNS 设置不同（通常是宽带运营商的 DNS，污染程度比移动数据轻）

**根本原因**：中国移动/联通/电信的**手机数据网络 DNS** 对 `*.github.io` 域名进行了污染，导致移动数据无法访问，而部分 WiFi 网络的 DNS 没有被污染，所以能正常访问。

解决方案建议：
这是一个**部署平台问题**，不是代码问题。要让中国大陆用户通过手机数据网络稳定访问，必须迁移到国内可访问的平台。

推荐方案对比：

| 方案 | 平台 | 费用 | 移动数据访问 | 操作难度 |
|------|------|------|-------------|----------|
| **方案A** | 阿里云 OSS + CDN | ~10元/月 | ✅ 最稳定 | 中等 |
| **方案B** | 腾讯云 COS + CDN | ~10元/月 | ✅ 最稳定 | 中等 |
| **方案C** | Cloudflare Pages | 免费 | ⚠️ 时好时坏 | 简单 |
| **方案D** | 自定义域名 + CDN | 需域名费 | ✅ 稳定 | 中等 |
